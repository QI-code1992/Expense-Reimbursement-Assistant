# 智能报销助手 Agent - 机会评估

## 文档信息

- 阶段：Stage 0 - Opportunity Validation
- 版本：v1.0
- 日期：2026-07-10
- 状态：已批准（GATE-001）
- 输入：当前对话中已确认的产品选择、官方市场资料复核

## 一、机会结论

建议进入需求定义阶段，但应以“中小企业轻量、聊天优先、员工确认后提交”为切入点，不在 MVP 中重建完整费控平台。

核心机会不是单纯做发票 OCR，而是把员工原本分散在上传、填单、查制度、补字段、反复退回中的工作，收敛为一段可解释、可确认、可追踪的会话流程。

## 二、目标问题

### 员工侧

- 不熟悉费用分类、公司制度和必填字段，填单成本高。
- 发票信息与业务用途分离，需要在多个页面重复录入。
- 退回原因通常出现得太晚，报销周期被拉长。
- 难以快速知道报销进度和下一步责任人。

### 审批人与财务侧

- 大量时间消耗在缺字段、错分类、抬头错误、疑似重复等低价值检查。
- 公司财务制度散落在 Word、PDF、通知和口头规则中，解释不一致。
- 传统系统更擅长流程承载，较少主动完成制度解释与补全追问。
- 中小企业对完整费控系统的实施和维护成本敏感。

## 三、已确认的产品方向

| 决策项 | 当前基线 |
|---|---|
| 首批客户 | 中小企业轻量版 |
| 员工入口 | 企业微信、钉钉或飞书机器人；具体首发平台待 Stage 1 确认 |
| 管理端 | 简单 Web 管理后台 |
| 首批费用 | 通用日常费用：餐饮、交通、办公用品、快递、招待等 |
| 审批方式 | 按金额分级审批 |
| 提交边界 | 必须由员工确认后提交 |
| 制度能力 | 财务制度 RAG 知识库 + 结构化规则引擎 |
| 产品方案 | 方案 1：聊天优先的轻量报销 Agent |

## 四、MVP 价值假设

### 用户价值

- 员工通过“发票 + 少量补充信息”完成提报，减少表单填写。
- Agent 在提交前指出问题并给出制度出处，降低退回率。
- 审批人与财务只处理需要判断的异常和业务合理性问题。

### 商业价值

- 以轻实施、轻管理后台切入中小企业，缩短部署和试用周期。
- 可按员工数、报销单量或企业订阅收费，具体模式留待商业验证。
- 后续可扩展差旅、预算、企业支付和 ERP/财务系统集成，但不纳入首版承诺。

## 五、MVP 产品边界

### 包含

- 员工通过聊天上传发票图片、PDF 或电子发票文件。
- 识别金额、日期、商户、发票号码、税号、购买方抬头等核心字段。
- 追问用途、项目/部门、费用类型、备注等缺失业务信息。
- 检查抬头、必填项、金额阈值、疑似重复票及制度约束。
- 生成报销单草稿，展示字段、风险提示和制度依据。
- 员工确认后提交，进入按金额分级审批。
- Web 后台支持报销管理、规则配置、员工管理、发票查看和基础统计。
- 状态通知、退回原因、操作日志和员工确认记录。

### 不包含

- 完整预算控制、借款、对公付款、银企直连和自动打款。
- 复杂差旅标准、行程预订与补贴计算。
- 深度 ERP/会计系统集成和长期电子会计档案。
- 无人值守审批或由大模型直接作最终合规裁决。
- 对发票真伪、税务抵扣资格作无依据的确定性承诺。

## 六、关键产品原则

1. 员工确认是提交的强制前置条件。
2. RAG 用于制度检索、解释、引用和软性判断，不代替硬规则与人工责任。
3. 高风险、低置信度或制度冲突必须升级给员工、审批人或财务确认。
4. Agent 不编造缺失业务事实，也不绕过审批规则。
5. 每次识别、修改、判断、确认和审批均应可追溯。

## 七、MVP 验证假设

| 假设 | 建议验证方式 | 初步通过标准 |
|---|---|---|
| 员工愿意用聊天完成报销 | 5-10 家目标企业访谈 + 可用性测试 | 多数受访者可独立完成核心流程 |
| 自动识别能显著减少录入 | 使用真实脱敏票据建立样本集 | 常见票据核心字段可用率目标在 Stage 1 统一 |
| 制度 RAG 能减少咨询与退回 | 以真实制度问答和违规样例评测 | 回答有出处；无法判断时不越权 |
| 按金额审批覆盖首批客户 | 访谈财务负责人并收集现行流程 | 试点客户主要流程无需定制开发 |
| 轻后台足够支撑试点 | 财务人员任务走查 | 能完成配置、审核、退回、确认和导出 |

## 八、成功指标候选

- 从上传票据到生成可提交草稿的中位耗时。
- 每张报销单需要员工手工补充/修改的字段数。
- 首次提交通过率与因资料问题导致的退回率。
- 制度问答的引用覆盖率、答案正确率和拒答准确率。
- 财务处理每单平均耗时。
- 试点企业激活率、周活跃报销人数和四周留存。

识别指标在此前讨论中出现过“80%+ 核心字段可用”和“95% OCR 目标”两种口径。Stage 1 必须按票据类型、字段和评测方法统一口径，当前不把任一数字视为正式承诺。

## 九、主要风险

- 票据类型、图像质量和电子发票格式差异导致识别效果波动。
- 公司制度文本含糊、冲突或缺少生效日期，RAG 可能检索正确但结论不唯一。
- 发票、员工和消费信息属于敏感财务数据，需要明确数据驻留、权限、加密、脱敏和保留策略。
- 企业微信、钉钉、飞书的能力、审核和部署方式不同，首发平台会影响周期。
- 伪造票据、重复报销和发票查验需要专门能力，不能只依赖大模型。
- 试点客户过早提出复杂差旅、预算或 ERP 集成，可能破坏轻量 MVP 边界。

## 十、Stage 1 待确认事项

- 首发 IM 平台及是否需要同时支持多个平台。
- 首批票据格式和费用类型的精确清单。
- 目标企业规模、月报销单量和并发假设。
- 财务制度样本文档、结构、版本和更新责任人。
- 审批层级、金额区间、财务复核位置和退回后流程。
- 私有化/SaaS 偏好、数据驻留、安全和合规要求。
- 试点时间、预算、团队与必须集成的现有系统。
- OCR、RAG、流程效率和业务结果的正式验收指标。

## 十一、Stage 0 建议

结论：`GO WITH CONDITIONS`。

`GATE-001` 已于 2026-07-10 获用户批准。本机会评估、MVP 边界和“RAG + 硬规则 + 人工兜底”的责任模型成为 Stage 0 v1.0 基线；Stage 1 将逐项关闭上述需求问题，并形成 PRD、SPEC、验收标准与追踪矩阵。
