# 可行性评估

## 文档信息

- 版本：v1.0
- 日期：2026-07-10
- 状态：已批准（GATE-001）
- 结论：技术可行，业务与合规边界需在 Stage 1 固化

## 一、总体判断

该产品可用成熟的票据识别、会话编排、检索增强生成、规则引擎、工作流和 Web 管理能力实现。主要难点不是单点模型能力，而是多种不确定性之间的责任编排：识别置信度、制度适用性、硬规则、员工确认、审批权限和审计证据必须形成一致链路。

建议采用“确定性流程为骨架，AI 为提取、理解和交互层”的实现原则。

## 二、能力分解

| 能力 | 可行性 | 主要约束 |
|---|---|---|
| 票据上传与解析 | 高 | 文件格式、大小、病毒扫描、重复上传、清晰度 |
| OCR/电子发票字段提取 | 高 | 不同票种和格式需分层评测；低置信度要人工确认 |
| 会话追问与草稿生成 | 高 | 必须基于显式状态机，避免重复追问和信息丢失 |
| 费用分类建议 | 中高 | 依赖企业科目配置、商户语义和历史数据；首版应允许纠正 |
| 财务制度 RAG | 中高 | 制度版本、适用范围、冲突条款和引用准确性是核心 |
| 硬规则检查 | 高 | 规则需结构化、可配置、可测试、可审计 |
| 疑似重复票 | 高 | 需稳定的发票唯一键与相似度策略；不能仅靠语义模型 |
| 发票真伪查验 | 待定 | 取决于合规可用的数据源或官方接口，不在首版默认承诺 |
| 分级审批 | 高 | 需要组织、金额区间、代理/离职/越级等边界定义 |
| Web 管理后台 | 高 | 角色权限、敏感字段展示、导出与操作日志是重点 |
| 多 IM 平台同时支持 | 中 | 平台 API、消息形态、文件下载与审核机制不同，建议首发单平台 |

## 三、RAG 与规则责任模型

### RAG 负责

- 回答“这笔费用是否符合制度、需要什么材料、为什么被提示”等问题。
- 检索并引用制度条款、版本、生效日期、适用部门或人员。
- 对自然语言制度给出建议性判断和补充材料提示。
- 在证据不足、条款冲突或问题超出制度时明确拒答或升级人工。

### 结构化规则负责

- 金额阈值与审批路径。
- 发票抬头、必填字段、费用类型和附件要求。
- 发票号码/代码等唯一键的重复检查。
- 角色、组织和数据访问权限。
- 可被系统确定执行的禁止项、时间边界和额度规则。

### 人员负责

- 员工确认业务事实与最终提交。
- 审批人判断业务必要性和例外申请。
- 财务确认合规、会计处理和付款前状态。
- 制度管理员发布、更新和废止制度版本。

## 四、概念性系统边界

1. IM 接入层：接收消息、附件和按钮操作，发送状态通知。
2. Agent 编排层：维护会话、任务状态、追问、确认和工具调用。
3. 文档智能层：文件解析、OCR、字段归一化、置信度和附件关联。
4. 制度决策层：RAG 检索、条款引用、结构化规则和风险分级。
5. 报销工作流层：草稿、确认、提交、审批、退回、财务确认和归档状态。
6. 管理后台：报销、发票、制度、规则、员工组织、统计和审计。
7. 基础设施层：身份权限、加密、日志、监控、任务队列和对象存储。

具体技术栈、云平台和部署架构应在 Stage 4 基于 Stage 1 的业务与非功能约束确定。

## 五、关键非功能风险

### 安全与隐私

- 发票可能包含姓名、电话、地址、税号、消费明细等敏感数据。
- 需要最小权限、传输/存储加密、租户隔离、敏感日志脱敏和下载控制。
- 数据保留期限、删除机制、模型供应商数据使用策略和跨境问题必须明确。

### 可靠性

- 文件处理和模型调用应支持幂等、超时、重试和失败补偿。
- 重复点击提交不得产生重复报销单。
- IM 平台不可用时，后台应保留任务状态并支持恢复或人工处理。

### 可解释与审计

- 每个风险提示应记录输入、规则/制度依据、版本、结果和人工修改。
- RAG 答案必须尽量绑定引用；无可靠依据时不得生成确定性结论。
- 员工确认、审批、退回和财务操作应形成不可抵赖的审计记录。

### 评测

- OCR 需按票据类型和字段统计准确率/可用率，而非只给总体数字。
- RAG 需评测检索命中、引用正确、答案正确、拒答和冲突处理。
- Agent 需评测任务完成率、追问轮次、工具调用成功率和越权率。

## 六、建议验证路径

### Phase A：需求与数据验证

- 访谈目标企业并收集脱敏流程、制度、票据和退回案例。
- 定义票据类型、字段字典、状态机、审批规则和制度责任模型。
- 建立小型黄金评测集及人工标准答案。

### Phase B：受控原型验证

- 先实现单 IM 平台、有限票种和一套示例制度。
- 打通上传、识别、补充、制度检查、草稿、确认、提交与审批闭环。
- Web 后台只覆盖试点所需配置、审核、查询、导出和日志。

### Phase C：试点验证

- 选择 1-3 家中小企业、有限部门和真实但受控的报销场景。
- 人工财务保留最终控制权，记录 Agent 建议与最终决定差异。
- 以处理耗时、修改字段数、退回率、制度问答正确率和满意度决定是否扩展。

## 七、Go/No-Go 条件

### Go

- 能获得合法、脱敏且有代表性的票据与制度样本。
- 试点客户接受员工确认、人工审批和财务兜底的责任模型。
- 单平台 MVP 可覆盖主要员工入口。
- 关键安全、数据驻留和模型使用约束可满足。

### No-Go 或降级

- 客户要求首版由 AI 自动完成最终审批或税务真实性担保。
- 无法获得制度/票据样本，却承诺高准确率和复杂规则覆盖。
- 首版必须同时覆盖复杂差旅、预算、支付和多个深度 ERP 集成。
- 数据合规要求与所选模型、部署或存储能力不兼容。

## 八、结论

技术结论为 `FEASIBLE WITH CONTROLLED SCOPE`。进入 Stage 1 时，应优先确定数据与制度样本、首发平台、状态与审批边界、非功能约束及量化验收口径；在这些输入明确前，不做技术栈和开发周期承诺。
