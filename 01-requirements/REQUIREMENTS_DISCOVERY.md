## 当前解释

- MVP 员工端优先实现飞书机器人会话，不承诺首版同时支持企业微信或钉钉。
- Web 管理后台继续服务财务、管理员及需要网页操作的审批场景。
- “选择飞书”目前只确定入口平台，不自动等同于使用飞书原生审批、飞书多维表格或其他飞书产品能力。
- 报销状态、审批链、金额分级、权限校验和操作日志由本产品维护；飞书机器人/交互卡片是审批操作入口，不使用飞书原生审批作为 MVP 主流程。
- 飞书卡片操作必须校验当前审批人和报销单状态，并防止重复点击造成重复审批。
- MVP 以多租户 SaaS 方式服务多家公司；企业组织、员工、制度、规则、发票、报销单、附件和审计数据必须按租户隔离。
- 租户管理员只能管理本企业数据，任何跨租户读取、检索、统计、导出或 Agent 上下文混用均禁止。
- 单租户托管和客户环境私有化部署不属于 MVP；具体租户隔离技术方案由 Stage 4 架构评审确定。
- MVP 容量验证按 1–3 个试点租户、每租户 50–500 名员工、每租户每月最多 3,000 张票据设计。
- 该规模是首轮试点及性能验证输入，不是产品未来的永久容量或商业套餐上限。
- P0 接收 JPG、PNG、PDF 和数电发票 XML；OFD 解析列为 P1，不作为 MVP 上线阻塞项。
- “核心字段准确率 ≥95%”按受支持、清晰票据中的字段值逐项统计，核心字段至少覆盖金额、开票日期、销售方、发票号码、购买方名称/抬头及可获得的税号字段。
- “可用草稿比例 ≥80%”指整张票据的核心字段无需员工修改即可进入报销草稿；员工补充用途、项目、部门等非票面业务信息不计为识别失败。
- 识别指标必须基于脱敏黄金样本集，按文件格式、票种和字段分别报告；低置信度、模糊、损坏或不支持的文件必须提示员工确认或重新上传。
- 制度知识生命周期为：草稿、解析完成、待复核、已发布、已替代或已停用；上传文件不得直接进入 Agent 判断。
- 租户财务管理员在发布前必须检查解析内容，并设置制度版本、生效日期、适用部门/人员、费用类型及其他适用范围。
- RAG 只检索当前业务场景可用的已发布版本；回答和风险提示必须返回制度名称、版本及条款出处。
- 新版本发布后，旧版本可标记为已替代；停用或替代不删除历史制度及其曾参与判断的审计记录。
- 制度冲突、适用范围不明确、未命中可靠条款或低置信度时，Agent 必须说明无法确定并升级人工处理。
- 模拟制度用于需求、交互和基础评测设计，必须明确标记为非真实企业制度，不得被解释为通用合规结论。
- 至少一份脱敏真实制度用于验证条款切分、适用范围、冲突处理、引用准确性和拒答行为；对应用户待办为 `UA-002`。
- 真实制度样本只放置在 Git 忽略的本地目录 `.private-input/policies/`，不得提交或推送到 GitHub。
- 每个租户可维护 2–3 个连续且不重叠的金额档位，并为各档位配置业务审批角色；直属主管固定为首级审批人。
- 所有业务审批完成后统一进入财务复核；自由拖拽流程设计器、任意条件分支和脚本规则不属于 MVP。
- 系统必须校验金额档位不存在空档或重叠；未完成有效审批配置的租户不得启用正式提交。
- 员工确认提交时保存审批配置版本和审批链快照；之后修改配置只影响新提交的报销单，不改写进行中的审批实例。
- 审批人或财务不通过时统一执行“退回修改”，必须填写退回原因；MVP 不提供审批人触发的 `REJECTED` 终止状态。
- 退回后原报销单号保留，单据回到员工可编辑状态；每次重提版本号递增，并保留退回原因、修改差异、确认记录和历史审批意见。
- 员工完成修改后必须重新确认提交，审批流程从首级审批人重新开始，不沿用此前节点的通过结果。
- 产品必须另行提供受控的撤回或关闭机制，避免不再报销的单据永久停留在可修改状态；具体权限待确认。
- 员工可在报销完成前撤回本人单据，终止状态为 `WITHDRAWN`；财务或租户管理员可关闭不再继续处理的单据，终止状态为 `CLOSED`，且必须填写关闭原因。
- `WITHDRAWN` 和 `CLOSED` 单据均转为只读并保留完整审计记录，不能恢复原审批流程；如需重新报销，只能复制为新草稿并生成新单号。
- 已终止且未实际报销的票据可用于新草稿，但必须关联原终止单据并展示历史使用提示；不得把终止记录误判为已完成的重复报销。
- 已完成报销的单据不可撤回或关闭；付款或完成状态的更正不属于 MVP，应由后续独立冲正流程处理。
