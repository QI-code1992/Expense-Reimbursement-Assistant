## 当前解释

- MVP 员工端优先实现飞书机器人会话，不承诺首版同时支持企业微信或钉钉。
- Web 管理后台继续服务财务、管理员及需要网页操作的审批场景。
- “选择飞书”目前只确定入口平台，不自动等同于使用飞书原生审批、飞书多维表格或其他飞书产品能力。
- 报销状态、审批链、金额分级、权限校验和操作日志由本产品维护；飞书机器人/交互卡片是审批操作入口，不使用飞书原生审批作为 MVP 主流程。
- 飞书卡片操作必须校验当前审批人和报销单状态，并防止重复点击造成重复审批。
- MVP 以多租户 SaaS 方式服务多家公司；企业组织、员工、制度、规则、发票、报销单、附件和审计数据必须按租户隔离。
- 租户管理员只能管理本企业数据，任何跨租户读取、检索、统计、导出或 Agent 上下文混用均禁止。
- 单租户托管和客户环境私有化部署不属于 MVP；具体租户隔离技术方案由 Stage 4 架构评审确定。
- MVP 容量验证按 1–3 个试点租户、每租户 50–500 名员工、每租户每月最多 3,000 张票据设计。
- 该规模是首轮试点及性能验证输入，不是产品未来的永久容量或商业套餐上限。
- P0 接收 JPG、PNG、PDF 和数电发票 XML；OFD 解析列为 P1，不作为 MVP 上线阻塞项。
- “核心字段准确率 ≥95%”按受支持、清晰票据中的字段值逐项统计，核心字段至少覆盖金额、开票日期、销售方、发票号码、购买方名称/抬头及可获得的税号字段。
- “可用草稿比例 ≥80%”指整张票据的核心字段无需员工修改即可进入报销草稿；员工补充用途、项目、部门等非票面业务信息不计为识别失败。
- 识别指标必须基于脱敏黄金样本集，按文件格式、票种和字段分别报告；低置信度、模糊、损坏或不支持的文件必须提示员工确认或重新上传。
- 制度知识生命周期为：草稿、解析完成、待复核、已发布、已替代或已停用；上传文件不得直接进入 Agent 判断。
- 租户财务管理员在发布前必须检查解析内容，并设置制度版本、生效日期、适用部门/人员、费用类型及其他适用范围。
- RAG 只检索当前业务场景可用的已发布版本；回答和风险提示必须返回制度名称、版本及条款出处。
- 新版本发布后，旧版本可标记为已替代；停用或替代不删除历史制度及其曾参与判断的审计记录。
- 制度冲突、适用范围不明确、未命中可靠条款或低置信度时，Agent 必须说明无法确定并升级人工处理。
- 模拟制度用于需求、交互和基础评测设计，必须明确标记为非真实企业制度，不得被解释为通用合规结论。
- 至少一份脱敏真实制度用于验证条款切分、适用范围、冲突处理、引用准确性和拒答行为；对应用户待办为 `UA-002`。
- 真实制度样本只放置在 Git 忽略的本地目录 `.private-input/policies/`，不得提交或推送到 GitHub。
- 每个租户可维护 2–3 个连续且不重叠的金额档位，并为各档位配置业务审批角色；直属主管固定为首级审批人。
- 所有业务审批完成后统一进入财务复核；自由拖拽流程设计器、任意条件分支和脚本规则不属于 MVP。
- 系统必须校验金额档位不存在空档或重叠；未完成有效审批配置的租户不得启用正式提交。
- 员工确认提交时保存审批配置版本和审批链快照；之后修改配置只影响新提交的报销单，不改写进行中的审批实例。
- 审批人或财务不通过时统一执行“退回修改”，必须填写退回原因；MVP 不提供审批人触发的 `REJECTED` 终止状态。
- 退回后原报销单号保留，单据回到员工可编辑状态；每次重提版本号递增，并保留退回原因、修改差异、确认记录和历史审批意见。
- 员工完成修改后必须重新确认提交，审批流程从首级审批人重新开始，不沿用此前节点的通过结果。
- 产品必须另行提供受控的撤回或关闭机制，避免不再报销的单据永久停留在可修改状态；具体权限待确认。
- 员工可在报销完成前撤回本人单据，终止状态为 `WITHDRAWN`；财务或租户管理员可关闭不再继续处理的单据，终止状态为 `CLOSED`，且必须填写关闭原因。
- `WITHDRAWN` 和 `CLOSED` 单据均转为只读并保留完整审计记录，不能恢复原审批流程；如需重新报销，只能复制为新草稿并生成新单号。
- 已终止且未实际报销的票据可用于新草稿，但必须关联原终止单据并展示历史使用提示；不得把终止记录误判为已完成的重复报销。
- 已完成报销的单据不可撤回或关闭；付款或完成状态的更正不属于 MVP，应由后续独立冲正流程处理。
- 金额分级审批按整张报销单中所有费用明细的申请金额合计计算，不按单笔最高金额单独升级审批。
- 员工首次提交及每次退回后重提时，系统均按当时合计金额重新匹配审批档位，并在员工确认前展示预计审批链。
- 若修改金额导致跨档，重提时使用新档位生成新的审批链快照，原版本匹配结果只作为历史审计记录保留。
- 金额区间边界必须唯一归属某一档位，后台配置和前端展示需明确包含/不包含关系，禁止边界金额匹配多个档位。
- 新租户不预填任何生产金额阈值；租户财务管理员必须完成金额档位、审批角色和财务复核人的配置，并明确确认启用。
- 审批配置未完成、校验失败或尚未启用时，员工可以上传票据和保存草稿，但不能正式提交报销单。
- 产品文档、测试数据和原型可使用模拟金额档位展示流程，但必须标记为“示例/模拟”，不得自动复制为租户生产规则。
- RAG 必须基于人工标注评测集验收，覆盖正常制度问答、疑似违规、制度版本与适用范围、冲突条款、无依据问题和跨租户隔离场景。
- 制度问答正确率不得低于 90%；条款引用正确率不得低于 95%；无依据、冲突或超范围问题的正确拒答/升级率不得低于 90%。
- 跨租户制度、检索片段、回答内容或上下文泄漏必须为 0；发现任何一例即为 P0 发布阻断，不得用总体平均指标抵消。
- 评测报告必须分别列出通过、失败、拒答、误答、错引、漏引和跨租户结果，并保存可复核的样例证据。
- MVP 在已确认试点规模下支持 100 名并发活跃用户；月可用性不得低于 99.5%，提前公告并经批准的维护窗口不计入。
- 不依赖外部 OCR/模型处理的普通 Web 和飞书机器人操作响应时间目标为 P95 不超过 2 秒。
- 文件上传后 3 秒内必须向用户返回“已受理”或明确失败；票据识别结果目标为 P95 不超过 30 秒，超时任务转为后台处理并在完成或失败时异步通知。
- 票据、RAG 和通知后台任务必须支持幂等、受控重试和失败状态查询，不得因重复回调或重试产生重复草稿、重复提交或重复审批。
- 性能与可用性测试报告必须注明环境、数据规模、并发模型、统计窗口、外部依赖是否计入及失败/超时数量。
- MVP 采用中国大陆境内的数据驻留和处理边界：生产环境中的票据附件、员工与组织信息、报销数据、制度文档、向量索引、业务日志及备份均须存储在中国大陆境内。
- OCR、LLM、向量化及其他可能接触租户数据的外部服务必须提供中国大陆境内的处理区域；MVP 不允许将生产租户数据发送至境外区域或境外模型端点。
- 外部服务商不得使用租户数据、提示词、检索片段、模型输入输出或衍生数据进行模型训练、产品改进、广告分析或其他二次利用。
- 与外部数据处理服务商的协议及配置必须明确处理目的、期限、方式、数据类别、保护措施、返还或删除要求，并保留可审计的供应商配置与调用证据。
- 任何跨境处理能力均不属于 MVP；未来如需引入，必须先发起正式变更请求，完成影响分析、租户开通机制和专项合规评审，不得通过普通运行时配置直接开启。
- 本项是当前产品的保守合规基线，不替代上线前由具备资质人员完成的法律与数据合规审查。
- MVP 定位为报销业务流程与审批证据系统，不定位为租户的法定电子会计档案系统，也不承诺法定会计档案的 30 年保管服务。
- 租户负责将需要归档的报销资料移交至其正式财务、会计或档案系统，并负责归档范围、保管期限、档案鉴定和销毁程序的最终合规性。
- 产品必须支持租户按权限导出报销单、原始票据附件、结构化字段、审批记录、员工确认记录、制度判断及引用、状态历史和必要审计元数据，确保资料可以被后续系统接收和复核。
- 导出结果应提供清单和完整性校验信息，但不得将“已导出”自动标记为“已完成法定归档”；归档成功状态如需回传，应在后续集成需求中另行定义。
- 租户停用服务或合同终止前必须提供受控的数据导出与交接窗口；交接完成后的保留和删除规则仍按待确认的数据生命周期要求执行。
- 产品界面、合同和管理员操作说明必须明确本产品与租户在电子会计档案管理方面的责任边界，避免将工作流留存误解为法定归档。
- 已提交报销单及其原始票据、业务字段、审批证据、制度判断和关键审计记录适用正式单据保留策略；租户可选择 1、3、5 或 10 年，默认 5 年。
- 正式单据在进入 `COMPLETED`、`WITHDRAWN` 或 `CLOSED` 等终态后开始计算保留期限；未结束的审批实例不得因达到日历期限而删除。
- 正式单据到期前必须通知租户管理员并支持完整导出；MVP 不在租户未确认的情况下自动删除正式单据。确认删除后记录删除申请、审批人、范围、时间和执行结果。
- 未提交且连续 90 天没有活动的草稿视为废弃草稿；删除前至少提前 7 天通知所有者，用户在通知期内继续编辑即可取消本次清理。
- OCR、模型调用和文件转换产生的中间文件、临时缓存及调试载荷应在任务完成或失败后 7 天内删除；原始票据、最终结构化结果和需随正式单据保留的证据不属于临时数据。
- 普通运行与诊断日志保留 1 年；能够证明提交、确认、审批、退回、撤回、关闭、导出或删除行为的关键审计记录随对应正式单据保留。
- 租户停用服务后提供 30 天受控导出窗口；窗口结束后删除在线生产数据，并在生产数据删除后的 90 天内完成备份轮转清除，期间备份不得恢复为日常可用业务数据。
- 因未决争议、安全调查、司法或监管要求需要暂停删除时，授权管理员可设置保留冻结；冻结必须记录原因、范围、授权人和解除时间，解除后恢复原生命周期规则。
- 所有删除任务必须按租户隔离、可审计、幂等且可验证，失败时进入告警和重试队列，不得误删其他租户或未到期数据。
- 员工只能主动删除本人尚未提交的草稿及其未被其他单据引用的附件；已提交、进入审批或终态的正式单据不得由员工删除。
- 租户管理员只能对已达到保留期限、未处于保留冻结且已完成完整导出的正式单据发起删除；进行中或未到期单据不得通过普通删除入口处理。
- 正式单据删除前必须要求租户管理员填写原因、重新验证身份，并在明确展示租户、时间范围、单据数量、附件数量及不可逆后果后进行二次确认。
- 删除请求受理后必须通知租户财务管理员；通知用于知情和监督，不构成第二审批节点，也不阻塞已满足条件的删除任务。
- 平台运营和客服人员无权独立发起、批准或扩大租户数据删除范围；只有在租户已授权的删除任务执行失败时，才可按原请求范围排查和恢复执行，并留下操作审计。
- 删除范围必须覆盖目标单据对应的业务数据、附件、搜索索引、向量索引及派生缓存；备份副本按既定 90 天轮转上限清除。
- 系统应向租户生成不包含已删除业务内容的删除结果回执，记录请求人、范围摘要、执行状态和时间；该回执按普通审计日志策略保留。
- MVP 数据导出采用标准 ZIP 证据包，通过 Web 管理后台异步生成；实时开放 API、Webhook 推送和特定财务软件直连不属于本项范围。
- 每张报销单应包含便于人工查阅的 PDF 摘要，展示单据基本信息、费用明细、审批过程、员工确认、制度判断及引用和当前终态。
- 证据包应包含 UTF-8 编码的 CSV 汇总表和明细表，以及用于机器处理的 JSON 数据；字段名称、数据类型、枚举值、时间格式和版本号必须有稳定定义。
- 原始票据附件必须保持原始文件内容不变，并通过安全文件名和映射清单关联到报销单及费用明细，不得因重命名破坏追溯关系。
- 根目录清单文件应记录导出任务、租户、筛选范围、生成时间、数据版本、文件路径、文件大小和每个文件的 SHA-256 摘要，支持接收方验证完整性。
- 导出任务必须支持按时间、状态及组织范围筛选，并在开始前展示预计数据范围；任务完成或失败后通知发起人，失败任务可在相同范围内安全重试。
- 导出生成、下载和失败重试均须执行租户隔离与权限校验并记录审计日志；下载入口必须受控且具有有效期，不得形成永久公开链接。
- 导出包生成成功不代表法定归档完成；租户仍须在接收系统中验证、导入并承担正式归档责任。
- Web 和飞书界面采用固定的角色与业务范围脱敏规则；员工可查看本人业务所需信息，审批人只能查看完成当前审批所需字段，财务和租户管理员也不得因角色名称而默认获得无范围限制的全量查看权限。
- 身份证件号码、个人手机号、个人邮箱、银行账户及其他非当前操作必需的个人标识在界面中默认部分遮蔽；需要查看原值的场景必须具备明确权限并记录访问审计。
- OCR、LLM、向量化和 RAG 调用只传递完成当前任务所需的最少数据；在不影响票据识别或制度判断时，应移除或替换员工账号、联系方式及其他无关个人标识。
- 应用日志、错误日志、链路追踪、监控事件和提示词调试记录不得写入原始票据文件、完整敏感字段、完整模型输入输出或可直接访问附件的永久链接。
- 完整证据包仅允许具备导出权限的财务管理员或租户管理员生成；发起导出时必须重新验证身份，并再次校验租户、组织范围和导出筛选条件。
- 导出包在服务器端及对象存储中必须加密保存，只能通过 HTTPS 和已登录会话下的短期受控链接下载；链接过期、权限撤销或租户停用后必须失效。
- 完整证据包不得直接作为飞书消息、飞书卡片或电子邮件附件发送；飞书和邮件只能发送不含敏感明文的状态通知，并引导用户回到受控 Web 页面下载。
- 导出文件属于临时派生数据，应在下载有效期结束后按临时数据策略自动清理；生成、查看下载入口、成功下载、失败和过期均记录审计事件。
- 脱敏、日志清洗、最小化模型输入和导出鉴权必须进入安全测试范围；发现跨租户数据或未经授权的敏感原文暴露时按 P0 发布阻断处理。
- 财务复核通过后，报销单进入非终态 `PENDING_PAYMENT`（待付款），不得仅因审批通过而标记为 `COMPLETED`。
- 实际资金支付在租户现有银行、网银或其他外部渠道完成；MVP 不发起付款、不连接银行或第三方支付平台，也不为付款目的采集员工银行卡号等非必要账户信息。
- 具备财务处理权限的用户须在 Web 后台登记付款日期、实付金额和备注；系统保存登记人、登记时间及确认记录，并通知员工处理结果。
- 实付金额必须与最终批准的报销金额一致后才能进入 `COMPLETED`；金额不一致时保持待付款并提示财务核实，MVP 不支持部分付款或多次分批付款。
- 付款登记提交前必须展示报销单号、员工、批准金额、付款日期和实付金额并要求二次确认；重复提交不得产生重复付款记录或重复完成通知。
- `COMPLETED` 的保留期限从付款登记确认完成时开始计算；误登记后的冲正、撤销付款或重新付款不属于 MVP，应按后续独立变更流程处理。
